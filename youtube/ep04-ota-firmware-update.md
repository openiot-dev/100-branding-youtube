# EP04. 이미 팔린 IoT 디바이스, 소프트웨어를 어떻게 고치지? — OTA 업데이트

## 메타 정보
- **예상 길이:** 5분
- **타겟:** IoT 제품의 사후 관리와 유지보수를 고민하는 기획자/경영진
- **키워드:** OTA 업데이트, 펌웨어 배포, 원격 업데이트, 롤백

---

## 인트로 (0:00 ~ 0:40)

여러분 스마트폰 업데이트 해보셨죠?
알림 뜨면 "업데이트" 누르고, 잠깐 기다리면 새 기능이 추가되고 버그가 고쳐집니다.

IoT 디바이스도 이게 됩니다. 이걸 **OTA(Over-The-Air)**, 무선 업데이트라고 합니다.

근데 스마트폰보다 **훨씬 어렵고 위험합니다.**

스마트폰은 업데이트 실패하면 서비스센터 가면 되지만,
고객 집 천장에 달린 IoT 센서가 업데이트 실패하면? 사다리 들고 가야 합니다.

오늘은 OTA를 **안전하게** 하는 방법을 알려드리겠습니다.

---

## 본론 1: OTA가 없으면 무슨 일이 벌어지나 (0:40 ~ 1:25)

OTA가 없는 IoT 디바이스를 상상해봅시다.

- 보안 취약점 발견 → **수거해서 고치고 재배송.** 1,000대면 비용이 어마어마합니다.
- 새 기능 추가 → **불가능.** 출시 때 기능 그대로 영원히 씁니다.
- 사소한 버그 → **고칠 방법 없음.** 고객 불만만 쌓입니다.
- 법규 변경 → **대응 불가.** 제품을 전량 리콜해야 할 수도 있습니다.

반대로 OTA가 있으면:
- 버그 발견 → 사무실에서 클릭 몇 번으로 전 세계 디바이스 동시 수정
- 새 기능 → 소프트웨어 업데이트만으로 제품 가치 향상
- 보안 패치 → 즉각 대응 가능

**OTA는 IoT 비즈니스의 생명줄**입니다.

---

## 본론 2: 업데이트는 어떤 과정으로 이루어지나 (1:25 ~ 2:40)

전체 과정을 택배 시스템에 비유해서 설명하겠습니다.

**1단계 — 소포 준비 (관리자)**
개발팀이 새 소프트웨어를 만들어서 창고(클라우드 저장소)에 올립니다.
웹 대시보드에서 "이 소프트웨어를 어떤 디바이스에 보낼지" 선택합니다.

배송 방식도 두 가지 중 선택:
- **스냅샷 배송:** 지금 등록된 디바이스에만 보냄
- **연속 배송:** 앞으로 새로 등록되는 디바이스에도 자동으로 보냄

**2단계 — 배송 알림 (클라우드 → 디바이스)**
클라우드가 해당 디바이스들에게 "새 소포 있어요, 받으세요" 메시지를 보냅니다.

**3단계 — 수령 및 설치 (디바이스)**
디바이스가 새 소프트웨어를 다운로드하고, 설치하고, 재시작합니다.

**4단계 — 수령 확인 (디바이스 → 클라우드)**
"잘 받았고, 정상 동작합니다!" 보고를 올립니다.

관리자는 대시보드에서 **각 디바이스가 어떤 상태인지** 실시간으로 볼 수 있습니다:
- 대기 중 → 다운로드 중 → 설치 중 → 완료 (또는 실패)

---

## 본론 3: 실패하면 어떡해? — 안전장치 이야기 (2:40 ~ 3:50)

OTA에서 가장 중요한 건 **"실패해도 괜찮은 구조"**를 만드는 겁니다.

**안전장치 1: 이중 저장소 (A/B 파티션)**

비유하면 이렇습니다. 디바이스 안에 **방이 두 개** 있다고 생각하세요.
- A방에서 현재 소프트웨어가 돌아가고 있습니다.
- 새 소프트웨어는 **B방에 설치**합니다.
- B방 소프트웨어가 정상이면 → B방으로 이사
- B방 소프트웨어에 문제가 있으면 → **A방에 원래 것이 그대로 있으니까** 돌아가면 됩니다.

절대로 원본을 덮어쓰지 않습니다. 이게 핵심입니다.

**안전장치 2: 검증**
다운로드한 파일이 중간에 깨지지 않았는지 확인합니다. 택배 상자를 열었는데 내용물이 파손됐으면 설치를 거부합니다.

**안전장치 3: 점진적 배포**
1,000대에 한꺼번에 보내지 않습니다.
먼저 **10대에만 보내서** 문제 없는지 확인하고, 괜찮으면 나머지에 보냅니다.
만약 10대에서 문제가 발생하면 나머지 990대는 무사합니다.

## 본론 4: 현실에서 흔히 겪는 문제들 (3:50 ~ 4:30)

**"업데이트 중에 전원이 나가면요?"**
A/B 파티션 구조 덕분에, 설치가 완료되기 전에 전원이 나가도 원래 소프트웨어로 부팅됩니다.

**"인터넷이 느린 환경이면요?"**
작은 조각으로 나눠서 다운로드합니다. 중간에 끊겨도 이어받기가 됩니다.

**"모든 디바이스가 동시에 다운로드하면 서버 터지지 않나요?"**
시간차를 두고 순차적으로 배포합니다. 급한 보안 패치가 아니면 며칠에 걸쳐 나눠 보냅니다.

**"이전 버전으로 되돌리고 싶으면요?"**
새 버전에 문제가 있으면 이전 버전을 다시 배포하면 됩니다. 그래서 이전 버전 소프트웨어를 항상 보관해두어야 합니다.

## 아웃트로 (4:30 ~ 5:00)

OTA는 IoT 제품을 **살아있는 제품**으로 만듭니다. 출시 후에도 계속 개선할 수 있으니까요.

핵심을 기억하세요: **이중 저장소, 파일 검증, 점진적 배포.** 이 세 가지 안전장치가 있어야 수만 대 디바이스를 안전하게 업데이트할 수 있습니다.
