# EP07. IoT 센서 데이터, 어디에 저장하고 비용은 어떻게 관리하지?

## 메타 정보
- **예상 길이:** 5분
- **타겟:** IoT 데이터 관리와 비용이 걱정되는 사업 기획자/PM
- **키워드:** 데이터 저장, 비용 관리, 자동 삭제, 시계열 데이터

---

## 인트로 (0:00 ~ 0:40)

온도 센서 하나가 1초마다 데이터를 보냅니다.
하루에 86,400건. 한 달이면 약 260만 건.

디바이스가 1,000대라면? 한 달에 **26억 건**입니다.

이 데이터를 전부 저장하면 비용이 어떻게 될까요?
반대로, 지우면 나중에 필요할 때 어떡하죠?

오늘은 IoT 데이터를 **똑똑하게 저장하고, 비용을 관리하는 방법**을 알려드리겠습니다.

---

## 본론 1: IoT 데이터는 뭐가 다른가 (0:40 ~ 1:25)

IoT 센서 데이터는 보통 데이터와 성격이 다릅니다.

**쏟아져 들어온다.**
쇼핑몰은 사람이 물건을 살 때만 데이터가 생깁니다. IoT는 디바이스가 **쉬지 않고** 데이터를 보냅니다. 읽는 것보다 쓰는 양이 압도적으로 많습니다.

**시간이 생명이다.**
"지금 온도 25도"가 중요하지, "3개월 전 온도 25도"는 별로 중요하지 않습니다. 최신 데이터일수록 가치가 높습니다.

**형태가 제각각이다.**
온도 센서는 숫자 하나를 보내고, 공기질 센서는 여러 수치를 보내고, 카메라는 이미지를 보냅니다. 데이터 형태가 디바이스마다 다릅니다.

이런 특성 때문에 일반적인 데이터베이스가 아니라 **IoT에 맞는 저장 방식**이 필요합니다.

---

## 본론 2: 데이터 저장소 선택 — 왜 DynamoDB인가? (1:25 ~ 2:20)

저장소를 고를 때 보통 세 가지 선택지가 있습니다.

**선택지 1: 전통적인 데이터베이스 (MySQL 등)**
엑셀 표처럼 정해진 칸에 데이터를 넣습니다. 깔끔하지만, 초당 수만 건 데이터를 넣으려면 버겁습니다. 그리고 센서 종류가 바뀔 때마다 표 구조를 바꿔야 합니다.

**선택지 2: 시계열 전용 데이터베이스 (InfluxDB 등)**
시계열 데이터에 최적화되어 있지만, 서버를 직접 운영해야 합니다. 관리 인력이 필요합니다.

**선택지 3: DynamoDB (이 플랫폼의 선택)**
- **관리 불필요.** AWS가 알아서 운영합니다.
- **자동 확장.** 데이터가 아무리 많아도 알아서 처리합니다.
- **유연한 구조.** 온도 센서든, 습도 센서든, 어떤 형태의 데이터든 그냥 넣으면 됩니다.
- **쓴 만큼만 과금.** 데이터가 적으면 비용도 적습니다.

초기 스타트업이라면 DynamoDB가 가장 현실적입니다. 서버 관리에 인력을 쓸 여유가 없으니까요.

---

## 본론 3: 데이터를 어떻게 정리하나 (2:20 ~ 3:15)

데이터가 들어오는 구조를 보겠습니다.

센서가 데이터를 보내면, 이런 식으로 저장됩니다:

```
디바이스 ID: 온도센서-001
시간: 2024년 3월 15일 14:30:22
데이터: 온도 25.3도, 습도 60.1%, 배터리 87%
```

그리고 검색은 주로 이런 식으로 합니다:
- "온도센서-001의 오늘 데이터 보여줘"
- "지난 1시간 동안의 온도 변화 보여줘"

시간순으로 저장되어 있으니까, "몇 시부터 몇 시까지"를 말하면 쭉 꺼내줄 수 있습니다.
도서관에서 책을 날짜순으로 꽂아두면, "3월 책 다 가져와"가 쉬운 것과 같습니다.

---

## 본론 4: 비용 관리의 핵심 — 자동 삭제(TTL) (3:15 ~ 4:10)

IoT에서 가장 큰 비용은 **저장 비용**입니다. 데이터가 계속 쌓이기만 하면 비용이 끝없이 증가합니다.

이 플랫폼은 **자동 삭제(TTL)** 기능을 씁니다.

비유하면 **편의점 유통기한 관리**와 같습니다.
우유에 유통기한이 찍혀 있잖아요? 기한이 지나면 자동으로 폐기됩니다.

이 플랫폼에서는 모든 센서 데이터에 **30일 유통기한**을 붙입니다.
30일이 지나면 데이터베이스가 자동으로 삭제합니다. 별도 작업이 필요 없습니다.

"30일 지난 데이터도 필요하면 어떡해요?"

좋은 질문입니다. 이럴 때는 **핫/콜드 분리 저장**을 합니다.

- **핫 저장소 (DynamoDB):** 최근 30일 데이터. 빠르게 조회 가능. 비용 높음.
- **콜드 저장소 (S3):** 30일 이후 데이터. 조회는 느리지만 비용이 1/10 수준.

마치 **자주 쓰는 서류는 책상 위에, 오래된 서류는 창고에** 보관하는 것과 같습니다.
창고가 훨씬 저렴하잖아요.

---

## 본론 5: 전체 데이터 흐름 한눈에 보기 (4:10 ~ 4:40)

디바이스에서 출발한 데이터가 어떻게 흘러가는지 정리하면:

```
센서 → 인터넷(MQTT) → 클라우드 수신 → 자동 분류
                                        ├→ 최근 데이터 → DynamoDB (30일 보관)
                                        ├→ 오래된 데이터 → S3 (장기 보관)
                                        └→ 이상 감지 시 → 알림 발송
```

핵심은 이 분류가 **자동으로** 이루어진다는 겁니다. 사람이 매번 "이 데이터는 어디에 넣어야지" 하지 않아도 됩니다. 규칙만 한 번 설정하면 클라우드가 알아서 처리합니다.

---

## 아웃트로 (4:40 ~ 5:00)

IoT 데이터 관리의 핵심은 세 가지입니다:
1. 쏟아지는 데이터를 버티는 저장소 선택
2. 자동 삭제로 비용 통제
3. 최근 데이터와 오래된 데이터 분리 보관

다음 영상에서는 디바이스가 인터넷이 끊겨도 명령을 받을 수 있는 신기한 기술, **Device Shadow**를 알려드리겠습니다.

구독과 좋아요, 부탁드립니다!
