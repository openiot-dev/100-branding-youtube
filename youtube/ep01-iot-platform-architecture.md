# EP01. IoT 플랫폼, 대체 어떤 구조로 만들어야 할까?

## 메타 정보
- **예상 길이:** 5분
- **타겟:** IoT 제품을 만들고 싶지만 어디서 시작할지 모르는 분
- **키워드:** IoT 아키텍처, 서버리스, AWS, 클라우드

---

## 인트로 (0:00 ~ 0:40)

여러분, 스마트 전구 써보셨죠?
앱에서 버튼 하나 누르면 집에 있는 전구가 켜집니다.

간단해 보이지만, 이 뒤에는 엄청난 시스템이 돌아가고 있습니다.

전구는 인터넷에 연결되어야 하고, 어딘가에 있는 서버가 명령을 전달해야 하고, 앱은 전구 상태를 실시간으로 보여줘야 합니다.

오늘은 이런 IoT 시스템을 **어떤 구조로 만드는지** 쉽게 설명해드리겠습니다.

---

## 본론 1: IoT 시스템의 3층 구조 (0:40 ~ 1:50)

IoT 시스템은 아파트처럼 **3층 구조**로 생각하면 됩니다.

**1층 — 디바이스 (현장 일꾼)**
실제 하드웨어입니다. 온도를 재고, LED를 켜고, 문을 잠급니다.
이 프로젝트에서는 ESP32라는 손톱만 한 칩을 씁니다. 가격이 몇천 원인데 Wi-Fi가 됩니다.

**2층 — 클라우드 (중앙 관제실)**
디바이스들이 보낸 데이터를 모아서 저장하고, 명령을 전달하는 두뇌 역할입니다.
우리가 직접 서버 컴퓨터를 사는 게 아니라, **AWS라는 클라우드 서비스를 빌려 씁니다.**
쉽게 말해 아마존이 운영하는 초대형 컴퓨터실을 월세 내고 쓰는 겁니다.

**3층 — 앱/웹 대시보드 (사용자 화면)**
사용자가 실제로 보는 화면입니다. 스마트폰 앱이나 웹 브라우저에서 디바이스를 모니터링하고 제어합니다.

핵심은 이 세 층이 **각자 역할이 명확하게 나뉘어 있다**는 겁니다.

---

## 본론 2: "서버리스"가 뭔데? 왜 쓰는 건데? (1:50 ~ 3:00)

여기서 "서버리스"라는 개념이 나옵니다.

**비유로 설명하면 이렇습니다.**

옛날 방식은 **식당을 직접 차리는 것**과 같습니다. 건물 계약하고, 주방 설비 사고, 직원 뽑고, 손님이 없어도 임대료 내야 합니다.

서버리스는 **공유 주방을 빌리는 것**입니다. 주문이 들어올 때만 주방을 쓰고, 안 쓸 때는 돈이 안 나갑니다. 손님이 갑자기 몰려도 주방이 자동으로 늘어납니다.

IoT에서 이게 특히 중요한 이유가 있습니다.

- 디바이스 100대일 때와 10만 대일 때 데이터양이 **1000배** 차이납니다. 서버리스는 자동으로 처리량을 늘려줍니다.
- 새벽 3시에 데이터가 거의 안 오면? **비용도 거의 0원**입니다. 서버를 24시간 켜놓으면 새벽에도 돈이 나가지만요.
- 서버 컴퓨터가 고장나면? **우리가 고칠 필요 없습니다.** AWS가 알아서 합니다.

3명짜리 스타트업이 수만 대 디바이스를 관리할 수 있는 비결이 바로 서버리스입니다.

---

## 본론 3: 실제로 뭘 쓰는지 한눈에 보기 (3:00 ~ 4:10)

너무 기술적으로 들어가지 않고, 역할 중심으로 정리해드리겠습니다.

| 역할 | 사용 기술 | 쉬운 비유 |
|------|-----------|-----------|
| 디바이스 | ESP32 + C언어 | 현장 센서/액추에이터 |
| 디바이스 ↔ 클라우드 통신 | MQTT (AWS IoT Core) | 디바이스 전용 메신저 |
| API 서버 | AWS Lambda | 주문 들어올 때만 켜지는 주방 |
| 데이터 저장 | DynamoDB | 자동으로 늘어나는 창고 |
| 사용자 인증 | AWS Cognito | 건물 출입 카드 시스템 |
| 웹 대시보드 | Next.js (React) | 관제실 모니터 화면 |
| 모바일 앱 | React Native | 스마트폰 리모컨 |
| 인프라 설정 | Terraform | 건물 설계 도면 (코드로 관리) |

핵심 포인트: **각각이 하나의 역할만 하고, 서로 독립적으로 동작합니다.**

전구 제어 기능에 문제가 생겨도, 센서 데이터 수집은 멀쩡하게 돌아갑니다.
마치 식당에서 주방에 불이 나도 계산대는 작동하는 것처럼요. (물론 식당은 문제가 되겠지만, 시스템에서는 이게 장점입니다.)

---

## 본론 4: 이 구조의 현실적인 장단점 (4:10 ~ 4:35)

**장점:**
- 적은 인원으로 대규모 서비스 가능
- 쓴 만큼만 비용 발생
- 서버 관리 부담 없음

**단점:**
- AWS에 의존하게 됨 (다른 곳으로 옮기기 어려움)
- 처음 배우는 데 시간이 걸림
- 문제가 생기면 여러 서비스를 왔다갔다 하며 원인을 찾아야 함

하지만 초기 IoT 스타트업이라면 **서버리스의 장점이 단점을 압도합니다.** 서버 관리할 인력도, 돈도 아끼면서 제품에 집중할 수 있으니까요.

---

## 아웃트로 (4:35 ~ 5:00)

정리하면, IoT 시스템은 **디바이스 - 클라우드 - 앱** 3층 구조이고, 서버리스로 만들면 적은 인원과 비용으로도 수만 대 디바이스를 관리할 수 있습니다.

다음 영상에서는 디바이스가 클라우드와 **어떻게 대화하는지**, IoT의 핵심 통신 방식을 쉽게 설명해드리겠습니다.

구독과 좋아요, 부탁드립니다!
