# EP02. IoT 디바이스는 어떻게 대화할까? - MQTT 쉽게 이해하기

## 메타 정보
- **예상 길이:** 5분
- **타겟:** IoT 통신 방식이 궁금한 비개발자/기획자
- **키워드:** MQTT, 통신 프로토콜, 실시간 제어, AWS IoT Core

---

## 인트로 (0:00 ~ 0:30)

스마트 전구에게 "켜져!"라고 명령을 보내면, 전구가 켜집니다.

근데 이 명령은 어떤 방식으로 전달되는 걸까요?
카카오톡처럼 메시지를 보내는 건가? 전화를 거는 건가?

오늘은 IoT 세계의 **공용어**라고 할 수 있는 MQTT에 대해 쉽게 설명해드리겠습니다.

---

## 본론 1: 인터넷 통신, 두 가지 방식 (0:30 ~ 1:30)

우리가 아는 인터넷 통신은 대부분 **"물어보면 대답하는"** 방식입니다.

웹 브라우저에서 네이버에 접속하면, 브라우저가 "네이버 메인페이지 줘"라고 요청하고, 서버가 "여기 있어"라고 응답합니다. 이걸 HTTP라고 합니다.

그런데 IoT에서는 이게 안 맞습니다. 왜?

- 센서가 **수시로** 데이터를 보내야 합니다. 매번 "보내도 돼?" 물어보는 건 비효율적이죠.
- 서버에서 디바이스로 **먼저 말을 걸어야** 합니다. "전구야, 꺼져!" 같은 명령을요.
- 디바이스가 **수만 대**입니다. 하나하나 전화 거는 방식은 감당이 안 됩니다.

그래서 IoT는 **"게시판 방식"**을 씁니다. 이게 MQTT입니다.

비유하면 이렇습니다. 회사 게시판에 "영업팀 공지"라는 게시판이 있으면, 영업팀 사람들이 그 게시판을 구독합니다. 누가 글을 올리면 구독자 전원에게 알림이 갑니다. 따로 전화할 필요 없이요.

MQTT도 같습니다. 특정 "채널"(토픽이라고 합니다)에 메시지를 올리면, 그 채널을 구독한 모든 곳에서 자동으로 받습니다.

## 본론 2: 토픽 — IoT 메시지의 주소 체계 (1:30 ~ 2:45)

MQTT에서 가장 중요한 건 **토픽(Topic)** 설계입니다. 토픽은 메시지의 주소 같은 겁니다.

이 플랫폼에서 쓰는 토픽 구조를 볼까요?

```
회사A / 1층사무실 / 온도센서1 / sensor
```

실제로는 이렇게 생겼습니다:
```
{회사ID}/{장소ID}/{디바이스ID}/{메시지종류}
```

메시지 종류는 네 가지입니다:

| 종류 | 방향 | 쉬운 설명 |
|------|------|-----------|
| `/status` | 디바이스 → 클라우드 | "나 지금 켜져 있어" (상태 보고) |
| `/command` | 클라우드 → 디바이스 | "LED 파란색으로 바꿔" (명령) |
| `/sensor` | 디바이스 → 클라우드 | "지금 온도 25도야" (센서 데이터) |
| `/ota` | 클라우드 → 디바이스 | "새 소프트웨어 업데이트 있어" |

이 주소 체계가 잘 만들어져 있으면, "1층 사무실의 모든 센서 데이터"를 한 번에 받는 것도 가능합니다. 주소에 와일드카드를 쓰면 됩니다. 마치 "서울시 강남구 *" 하면 강남구 전체 우편물을 다 받는 것처럼요.

## 본론 3: 왜 직접 안 만들고 AWS IoT Core를 쓸까? (2:45 ~ 3:45)

MQTT 시스템을 직접 만들 수도 있습니다. Mosquitto라는 무료 소프트웨어가 있거든요.

근데 이건 마치 **우체국을 직접 운영하는 것**과 같습니다.
건물 임대하고, 우체부 고용하고, 편지가 몰리면 우체부를 더 뽑고...

**AWS IoT Core**는 **아마존이 운영하는 초대형 우체국을 빌리는 것**입니다.

장점이 뭐냐면:

1. **규모 걱정 없음.** 디바이스가 10대든 100만 대든, 아마존이 알아서 처리합니다.
2. **보안이 기본 내장.** 각 디바이스에게 전용 신분증(인증서)을 발급하고, 모든 통신을 암호화합니다. 우리가 보안 전문가를 따로 고용할 필요가 없습니다.
3. **접근 제어.** "이 디바이스는 이 채널에만 글을 올릴 수 있어"라는 규칙을 세밀하게 설정할 수 있습니다. 해커가 디바이스 하나를 탈취해도 다른 디바이스에 영향을 줄 수 없습니다.
4. **자동 연결 기능.** 메시지가 들어오면 자동으로 데이터베이스에 저장하거나, 알림을 보내는 규칙을 설정할 수 있습니다. 별도 프로그래밍 없이요.

## 본론 4: MQTT가 가벼운 이유 — 배터리와의 관계 (3:45 ~ 4:30)

"그냥 인터넷 통신(HTTP) 쓰면 안 돼?"라고 물어보실 수 있습니다.

핵심 차이는 **가벼움**입니다.

HTTP로 메시지를 보내면, 뒤에 붙는 부가 정보(헤더)가 수백 글자입니다.
"보내는 사람, 받는 사람, 시간, 형식, 쿠키, 인증 정보..." 편지보다 봉투가 더 큰 셈이죠.

MQTT의 부가 정보는 **겨우 2글자**입니다. 편지만 딱 보내는 겁니다.

이게 왜 중요하냐면, IoT 디바이스 중 상당수는 **배터리로 동작**합니다.
데이터를 보낼 때마다 전력을 쓰는데, 보내는 양이 적을수록 배터리가 오래 갑니다.
센서가 하루에 만 번 데이터를 보낸다고 생각해보세요. 매번 수백 글자 vs 2글자, 배터리 수명 차이가 엄청납니다.

또 하나, MQTT는 **한 번 연결하면 계속 유지**합니다.
HTTP는 매번 "전화 걸고 → 대화 → 끊고"를 반복하는데,
MQTT는 "전화 한 번 걸어놓고 계속 대화"합니다. 연결을 새로 맺는 에너지를 아끼는 거죠.

---

## 아웃트로 (4:30 ~ 5:00)

정리하면, MQTT는 IoT의 **게시판 메신저**입니다.
가볍고, 양방향이고, 수만 대가 동시에 쓸 수 있습니다.
AWS IoT Core를 쓰면 이 시스템을 직접 운영하지 않고 바로 사용할 수 있고요.

다음 영상에서는 디바이스를 박스에서 꺼내서 전원만 넣으면 자동으로 클라우드에 연결되는 마법 같은 과정, **프로비저닝**을 알려드리겠습니다.

구독과 좋아요, 부탁드립니다!
