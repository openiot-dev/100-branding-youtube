# EP10. 코드 한 줄 바꾸면 자동으로 배포된다 — IoT 플랫폼 자동화

## 메타 정보
- **예상 길이:** 5분
- **타겟:** IoT 플랫폼 운영 효율화에 관심 있는 PM/개발팀 리드
- **키워드:** CI/CD, 자동 배포, 자동 테스트, 릴리스 자동화, GitHub Actions

---

## 인트로 (0:00 ~ 0:40)

개발자가 코드를 한 줄 수정했습니다.

옛날 방식이라면:
1. 코드를 압축해서
2. 서버에 접속해서
3. 파일을 올리고
4. 서비스를 재시작하고
5. 잘 되는지 확인하고...

이 과정에서 실수 한 번이면 서비스가 다운됩니다.

이 플랫폼은 다릅니다. 코드를 수정하면 **자동으로 검증하고, 자동으로 배포합니다.**
사람 손을 최소화하면 실수도 최소화됩니다.

이걸 **CI/CD**라고 합니다. 오늘 쉽게 설명해드리겠습니다.

---

## 본론 1: CI/CD가 뭔가요? (0:40 ~ 1:30)

CI/CD는 두 가지를 합친 말입니다.

**CI (지속적 통합 — Continuous Integration)**
개발자가 코드를 올릴 때마다 **자동으로 검사**하는 것.
비유하면 **편집부의 교정 담당**입니다. 원고를 제출하면 맞춤법, 문법, 일관성을 자동으로 체크하고, 문제가 있으면 "이거 고치세요"라고 돌려보냅니다.

**CD (지속적 배포 — Continuous Deployment)**
검사를 통과하면 **자동으로 실제 서비스에 반영**되는 것.
교정이 끝난 원고가 자동으로 인쇄소에 넘어가서 출판되는 것과 같습니다.

사람이 하는 일: 코드를 작성해서 올리기.
나머지(검사, 빌드, 배포, 확인)는 전부 자동입니다.

---

## 본론 2: 자동 검사 — 나쁜 코드 걸러내기 (1:30 ~ 2:30)

개발자가 코드를 올리면 **세 가지 자동 검사**가 돌아갑니다.

**검사 1: 변경 기록 형식 확인**
코드를 올릴 때 "무슨 변경인지" 메모를 남기는데, 이 메모 형식이 정해져 있습니다.

```
feat: 새로운 기능 추가
fix: 버그 수정
```

왜 형식을 강제할까요? 나중에 "이번 달에 어떤 기능이 추가되고 어떤 버그가 고쳐졌는지"를 **자동으로 정리**할 수 있기 때문입니다. 사람이 일일이 정리하지 않아도 됩니다.

**검사 2: 코드 스타일 확인**
띄어쓰기, 줄바꿈 같은 코드 스타일이 팀 규칙에 맞는지 자동 확인합니다.
10명이 10가지 스타일로 코드를 쓰면 읽기가 너무 어려우니까요.

**검사 3: 빌드 확인**
코드가 에러 없이 정상적으로 만들어지는지 확인합니다.
"내 컴퓨터에서는 됐는데..."를 방지합니다.

세 가지 중 **하나라도 실패하면 코드가 반영되지 않습니다.** 문제가 있는 코드가 서비스에 섞이는 걸 원천 차단하는 겁니다.

---

## 본론 3: 자동 배포 — 바뀐 부분만 골라서 (2:30 ~ 3:40)

검사를 통과하면 자동 배포가 시작됩니다.

이 플랫폼은 여러 부분으로 이루어져 있습니다:
- 웹 대시보드 (사용자 화면)
- API 서버 (데이터 처리)
- 인프라 설정 (서버/데이터베이스 설정)
- 자동화 서비스 (부가 기능)

핵심: **바뀐 부분만 배포합니다.**

웹 화면만 고쳤으면? 웹 대시보드만 새로 배포. API 서버는 건드리지 않음.
API 코드를 고쳤으면? API 서버만 업데이트. 웹 화면은 그대로.

이걸 자동으로 판단합니다. "어떤 파일이 바뀌었나" 확인해서, 해당하는 서비스만 골라 배포하는 겁니다.

비유하면 **아파트 보수 공사**와 같습니다.
301호 수도관이 고장났으면 301호만 공사합니다. 302호까지 같이 공사하지 않잖아요.

그리고 배포가 끝나면 **"오늘 뭐가 배포됐는지" 요약 보고서**가 자동으로 생성됩니다.

---

## 본론 4: 릴리스 자동화 — 버전 관리도 자동 (3:40 ~ 4:30)

소프트웨어에는 버전이 있습니다. v1.0, v1.1, v2.0 같은 거요.

보통은 사람이 "이번 버전은 1.3으로 하자" 하고 정합니다.
이 플랫폼은 이것도 자동입니다.

변경 기록 메모를 분석해서:
- "새 기능 추가" 메모가 있으면 → 1.2 → **1.3**으로 올림
- "버그 수정" 메모만 있으면 → 1.3.0 → **1.3.1**로 올림

그리고 자동으로:
- **변경 내역서(CHANGELOG)** 작성
- **릴리스 노트** 생성
- 심지어 릴리스 노트를 바탕으로 **블로그 글까지 자동 생성**

맞습니다. AI가 릴리스 노트를 읽고, "이번 업데이트에서는 이런 기능이 추가되고 이런 버그가 수정되었습니다"라는 블로그 글을 자동으로 만들어서 게시합니다.

개발자는 코드만 쓰면 됩니다. 버전 관리, 문서 작성, 블로그 포스팅까지 전부 자동이에요.

---

## 본론 5: 이 자동화가 사업에 미치는 영향 (4:30 ~ 4:50)

**속도:** 코드 수정 후 30분 이내에 실제 서비스에 반영. 수동이면 반나절.
**안정성:** 자동 검사 덕분에 오류가 있는 코드가 서비스에 올라가지 않음.
**인력:** 배포 담당자가 따로 필요 없음. 개발자 3명이면 충분.
**투명성:** 언제, 뭐가, 왜 바뀌었는지 자동 기록. 감사 대응도 수월.

IoT 플랫폼은 관리해야 할 것이 많습니다 — 웹, API, 인프라, 디바이스까지.
자동화 없이 이걸 수동으로 하면 팀이 배포에 파묻혀서 제품 개발을 못 합니다.

---

## 아웃트로 (4:50 ~ 5:10)

이 10편의 시리즈를 통해 IoT 플랫폼의 A부터 Z까지 다뤘습니다.

1편: 전체 구조
2편: 디바이스 통신(MQTT)
3편: 초기 설정(프로비저닝)
4편: 원격 업데이트(OTA)
5편: 다수 고객 서비스(멀티테넌시)
6편: 보안
7편: 데이터 저장/관리
8편: 디바이스 제어(Device Shadow)
9편: 디바이스 내부 소프트웨어
10편: 자동 배포(CI/CD)
