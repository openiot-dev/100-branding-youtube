# EP09. IoT 디바이스 안에서는 무슨 일이 벌어질까? — ESP32 펌웨어 이야기

## 메타 정보
- **예상 길이:** 5분
- **타겟:** IoT 하드웨어/펌웨어 개발을 시작하려는 비전공자/기획자
- **키워드:** ESP32, 펌웨어, 마이크로컨트롤러, 동시 처리, 부팅

---

## 인트로 (0:00 ~ 0:40)

스마트 전구 안에 아주 작은 컴퓨터가 들어있다는 거 아시나요?

이 작은 컴퓨터가 동시에 하는 일이 놀랍습니다:
- Wi-Fi 연결 유지
- 클라우드에서 명령 수신
- 센서 데이터 읽기
- LED 제어
- 소프트웨어 업데이트 감시
- 자기 상태 보고

이걸 **손톱만 한 칩**에서 합니다. 메모리는 스마트폰의 1/10,000 수준이고요.

오늘은 이 작은 칩 안에서 **무슨 일이 벌어지는지** 쉽게 풀어드리겠습니다.

## 본론 1: ESP32가 뭔가요? (0:40 ~ 1:25)

ESP32는 IoT 세계의 **국민 칩**입니다.

크기는 우표만 하고, 가격은 몇천 원인데, 이 안에:
- **컴퓨터(CPU)** 2개 (동시에 두 가지 일 가능)
- **Wi-Fi** 내장 (별도 부품 불필요)
- **블루투스** 내장
- **센서 연결 핀** 다수

가 다 들어있습니다.

자동차로 치면 경차인데 **에어컨, 네비, 블루투스가 기본 탑재**된 느낌입니다.

전 세계 IoT 제품의 상당수가 이 칩을 씁니다. 그만큼 자료도 많고, 문제가 생겼을 때 해결책을 찾기 쉽습니다.

이 플랫폼은 ESP32에 **C 언어**로 소프트웨어를 작성합니다. ESP-IDF라는 공식 개발 도구를 사용하고요.

---

## 본론 2: 전원을 넣으면 무슨 일이 벌어지나 — 부팅 과정 (1:25 ~ 2:30)

디바이스에 전원을 넣는 순간부터 동작하기까지의 과정을 따라가 봅시다.

**1단계: 기억 확인**
"나 Wi-Fi 비밀번호 알고 있나?" 확인합니다.
디바이스 안에는 **전원이 꺼져도 안 지워지는 메모리**가 있습니다. 여기에 Wi-Fi 정보, 인증서 같은 중요한 정보가 저장되어 있습니다.

**2단계: Wi-Fi 비번 모르면?**
이전 영상에서 설명한 **SoftAP 모드**로 들어갑니다. 디바이스가 직접 Wi-Fi 공유기가 되어서 사용자에게 비밀번호를 물어봅니다.

**3단계: Wi-Fi 비번 알면?**
바로 Wi-Fi에 연결합니다. 그리고 인증서가 있는지 확인합니다.

**4단계: 인증서 없으면?**
클라우드에 접속해서 자동으로 인증서를 발급받습니다. (프로비저닝)

**5단계: 인증서 있으면?**
클라우드에 연결하고, 본격적으로 일을 시작합니다.

핵심은 **상황에 따라 다른 길을 간다**는 겁니다.
처음 켜는 디바이스와, 이미 설정된 디바이스의 경로가 다릅니다.
마치 신입 사원의 첫 출근과 재직자의 일반 출근이 다른 것처럼요.

## 본론 3: 한 번에 7가지 일을 어떻게 하나 — 멀티태스킹 (2:30 ~ 3:40)

ESP32가 동시에 처리하는 일이 7가지나 됩니다. 이걸 어떻게 할까요?

비유하면 **작은 식당의 사장님**과 같습니다.
주방에서 요리하면서, 손님 주문 받고, 전화도 받고, 배달 앱도 확인합니다.
실제로 동시에 하는 건 아니고, **빠르게 번갈아 가면서** 하는 거죠.

ESP32도 마찬가지입니다. "태스크"라는 단위로 일을 나누고, 아주 빠르게 번갈아 처리합니다.

| 태스크 | 하는 일 | 중요도 |
|--------|---------|--------|
| 클라우드 통신 | 메시지 주고받기 | 최우선 |
| 센서 읽기 | 온도, 습도 등 측정 | 높음 |
| 명령 실행 | LED 켜기/끄기 등 | 높음 |
| 업데이트 감시 | 새 소프트웨어 있나 확인 | 보통 |
| 상태 보고 | "나 살아있어" 주기적 보고 | 보통 |
| Wi-Fi 감시 | 연결 끊기면 재연결 | 높음 |
| 안전 장치 | 시스템 멈추면 자동 재부팅 | 최최우선 |

**중요도**가 핵심입니다. 클라우드 통신이 센서 읽기보다 우선입니다. 통신이 끊기면 아무리 센서를 읽어도 소용없으니까요.

그리고 가장 위에 **안전 장치(Watchdog)**가 있습니다. 시스템이 먹통이 되면 자동으로 재부팅하는 비상 버튼이에요. IoT 디바이스는 사람이 옆에 없으니까, 스스로 복구할 수 있어야 합니다.

## 본론 4: 메모리 — 아주 적은 공간에서 살아남기 (3:40 ~ 4:20)

여기서 놀라운 사실. ESP32의 메모리는 약 **512KB**입니다.

이게 얼마나 적으냐면:
- 스마트폰 메모리: 8,000,000KB (8GB)
- ESP32 메모리: 512KB
- **스마트폰의 16,000분의 1**입니다.

이 좁은 공간에서 7개 태스크가 동시에 돌아야 합니다.

그래서 메모리 관리가 생명입니다:
- 각 태스크에 **꼭 필요한 만큼만** 공간 배정
- 클라우드 통신은 암호화 때문에 **공간을 많이 차지** (약 50KB)
- 메모리 누수(쓰고 반납 안 하는 것)가 있으면 며칠 뒤 디바이스가 멈춤
- 주기적으로 "메모리 얼마나 남았나" 체크하는 로직 필요

마치 원룸에서 살면서 수납 공간을 아주 효율적으로 써야 하는 것과 같습니다.
큰 집(서버)에서는 대충 둬도 되지만, 원룸(ESP32)에서는 1cm가 아쉽습니다.

---

## 본론 5: 소프트웨어 구조 — 어떻게 정리하나 (4:20 ~ 4:45)

이 펌웨어는 기능별로 **파일을 깔끔하게 나눠놨습니다.**

| 파일 | 담당 |
|------|------|
| main.c | 총괄 지휘 (부팅, 태스크 생성) |
| wifi_manager.c | Wi-Fi 연결 담당 |
| mqtt_aws.c | 클라우드 통신 담당 |
| provision.c | 첫 등록(프로비저닝) 담당 |
| device_control.c | LED, 센서 제어 담당 |
| ota_update.c | 소프트웨어 업데이트 담당 |
| storage.c | 기억 저장소 관리 담당 |
| state.c | 전체 상태 관리 담당 |

각 파일이 하나의 역할만 합니다. 식당에서 주방장, 홀 매니저, 배달 담당이 분리된 것처럼요. 그래야 한 곳을 고칠 때 다른 곳이 영향받지 않습니다.
